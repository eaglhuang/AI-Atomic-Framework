# ATM First Principles

**Schedule version**: 1.0.0
**Status**: Normative Schedule A incorporated by AtomicCharter 2.1.0
**Effective date**: 2026-07-14
**Applies to**: ATM framework design, implementation, operation, and release acceptance

## 1. Authority and Purpose

AtomicCharter is ATM's sole constitutional authority and waiver entry point. This
schedule is ATM's highest design and acceptance guidance because the Charter
incorporates it. It cannot override a Charter invariant. A conflict with the
Charter fails closed until resolved through the Charter amendment process.

ATM governance exists only to protect delivery outcomes that would otherwise be
at material risk. A command, role, gate, artifact, or handoff has no inherent
value. It must protect at least one invariant in Section 3, and its measured cost
must remain within the budgets in Sections 5 and 6.

The words MUST, MUST NOT, SHOULD, and MAY are normative.

## 2. Definitions

- **Equivalent work**: the same starting revision, requested outcome, acceptance
  contract, model/provider class, environment, and required quality level.
- **Necessary validation**: the smallest sound validator set that covers the
  changed obligations, with a conservative fallback when coverage is uncertain.
- **Canonical evidence**: an ATM-wrapper-produced, content-bound receipt accepted
  by a hard gate. Agent-authored summaries are advisory, not canonical evidence.
- **Contribution**: a worker-owned set of file blobs based on a declared base
  revision and delivered through a manifest, not an unbounded shared-tree edit.
- **Closer**: the single actor authorized to assemble, seal, commit, and queue a
  push for a delivery unit.
- **Governed completion**: a clean, recoverable delivery that satisfies the same
  acceptance contract used by its comparison baseline.
- **Total tokens**: all input, output, cached-input charges when observable, tool
  prompts, captain work, worker work, retries, handoffs, routing, and recovery.
- **End-to-end time**: elapsed wall-clock time from accepted request to governed
  completion, including queueing, validation, retries, close, commit, and cleanup.

## 3. The Four Protected Invariants

### FP-001: Contribution ownership

An agent MUST NOT mutate or deliver files outside its owned contribution scope.
The captain MAY expand scope before the new file is mutated. Expansion MUST create
a new scope epoch, check current assignments and contributions for overlap, and
record the reason. If another contribution owns the file, work MUST queue or move
to the composer/closer. In-flight ownership transfer is forbidden.

ATM MUST use proposal-first contributions on one canonical worktree/base/HEAD.
Workers may use ephemeral non-Git proposal carriers, but they do not create a
second development worktree or branch. A worker submits bounded file/atom/anchor
intents and proposals; a neutral steward is the only writer that applies an
accepted shared-file composition to the canonical worktree.

### FP-002: Necessary validation only

A delivery MUST pass every validation obligation caused by its final payload. ATM
MUST NOT run validators merely because they exist or because an unrelated gate
historically ran them. Coverage starts with a reviewable path-to-validator map and
a conservative package or directory fallback.

A non-blocking full-suite canary MUST periodically validate an exact sealed commit
in a clean checkout. Canary failures MUST create mapping-gap feedback. Symbol-level
minimization MAY replace coarse mapping only after the canary demonstrates that it
is sound.

### FP-003: Small, automatic, content-bound evidence

Canonical evidence MUST be generated by the ATM wrapper that executes the
validator. Agents may reference a receipt ID but MUST NOT construct or edit a
canonical receipt. A receipt MUST bind the validator identity and version, command,
environment class, base identity, payload or tree digest, relevant scope, result,
and duration.

Evidence MUST be the smallest receipt sufficient to reproduce and audit the gate.
Logs and summaries are supporting diagnostics, not a substitute for digest-bound
evidence. Until real validator read sets exist, invalidation MUST be conservative.

### FP-004: One serialized closer

Exactly one closer may assemble, commit, and push a delivery unit. Concurrent
workers MAY prepare contributions, but they MUST NOT independently commit or push
the shared branch. The closer is protected by a diagnosable TTL mutex or an
equivalent queue.

The sealed Git commit is recovery truth. Runtime publication receipts are caches
that MUST be reconstructable idempotently from commit trailers and committed
payload/evidence digests. A crash MUST converge to either active work or a closed
delivery without a third ambiguous state.

## 4. Minimal Governance Architecture

ATM MUST prefer the smallest mechanism that preserves Section 3:

1. Route work as Single Agent unless independent contribution groups justify Team.
2. Use bounded non-Git proposal carriers for Team contributions and focused
   validation; keep the canonical worktree as the single development substrate.
3. Let an ATM wrapper create content-addressed receipts.
4. Assemble all accepted contributions at one barrier.
5. Derive final validation obligations from the assembled payload.
6. Let one closer seal, commit, publish, and queue the push.

File leases are scheduling hints in a proposal-first design. They prevent
duplicate effort but are not a reason to treat a shared physical file as a lock
or build a second content-merging system. Content identity, bounded logical
overlap, and adapter mergeability are rechecked at assembly.

Batch mode is a parent delivery queue, not a second task lifecycle. It MUST reuse
the same contribution, receipt, validation, and closer contracts. Only the queue
head may close, and checkpoint, closure, commit, claim release, and queue advance
MUST converge through one recoverable transaction boundary.

## 5. ATM Governance Cost Budget

For equivalent work, ATM MUST satisfy both independent ratios:

```text
ATM total tokens / no-ATM total tokens <= 1.10
ATM end-to-end time / no-ATM end-to-end time <= 1.10
```

The release objective for each ratio is `<= 1.05`. The ratios MUST NOT be averaged,
traded against one another, or improved by weakening validation or acceptance.

A feature without paired measurements MUST report `measurement-incomplete` and
remain advisory or shadow-only when it could increase cost. Existing mandatory
paths that predate telemetry MUST receive a measured remediation plan; missing
telemetry is not evidence of compliance.

## 6. Team Agents Acceleration Budget

Team Agents exist to accelerate delivery, not to demonstrate concurrency. Against
an ATM Single Agent baseline for equivalent work, Team mode MUST satisfy both:

```text
Team total tokens / Single Agent total tokens <= 0.50
Team end-to-end time / Single Agent end-to-end time <= 0.50
```

ATM MUST dynamically select the smallest useful roster. Team admission requires at
least two disjoint contribution groups and no unresolved shared bottleneck file on
the critical path. Historical measurements SHOULD reserve headroom before launch;
a predicted ratio of `<= 0.45` is the default promotion threshold.

If either measured ratio exceeds `0.50`, ATM MUST reduce the roster or fall back to
Single Agent for that work class. It MUST also write a structured efficiency
incident containing the task class, roster, dependency shape, token and time
ratios, queue/validation/close breakdown, reason codes, and next bounded experiment.
The optimization objective remains open until a strategy meets both ratios, but
ATM MUST NOT charge the user for unbounded automatic retries.

## 7. Configurable Atomicity Bound

ATM MUST expose one authoritative `atomization.maxLines` parameter. Its default is
`600`; repositories MAY lower it. Every newly extracted atom, map, script, or
supporting source module MUST remain at or below the active limit. A temporary
exception requires an explicit expiry, owner, and follow-up card.

Line count is a guardrail, not the decomposition model. Boundaries MUST preserve a
single responsibility, explicit inputs and outputs, testability, and stable
atom/map semantics. Existing oversized modules require a measured migration plan;
new extraction MUST NOT create another oversized facade behind a smaller wrapper.

## 8. Measurement Integrity

Comparisons MUST pin the base revision, workload fixture, model/provider class,
environment, cache policy, validator obligations, and completion definition. Token
usage from provider APIs is canonical when available. Estimates MUST be labeled
and cannot prove a release threshold.

Time reports MUST separate routing, queue wait, worker execution, validation,
assembly, close/commit, push, and recovery. Failed attempts and repair work remain
inside the totals. A fast result that leaves residue, an unrecoverable close, or
weaker evidence is not a completed sample.

ATM SHOULD optimize p50 and p95, but every promoted path MUST meet the hard ratios
on the defined acceptance cohort. Measurements and reason-coded failures are
product inputs and MUST feed future routing decisions.

## 9. Progressive Enforcement

New governance behavior MUST first run in proposal/shadow mode against the
current canonical path. This shadow mode is a non-Git comparison or proposal
carrier, not a detached development worktree. Promotion requires artifact parity,
crash consistency, no weaker validation, and compliance with the applicable cost
budgets. Rollback MUST select the last known canonical path without rewriting
user work.

When a first principle is not yet machine-enforced, documentation and telemetry
MUST say so plainly. Design intent MUST NOT be represented as implemented safety.

## 10. Amendment

Changes to this schedule require the paired AtomicCharter amendment procedure. A
change that alters a Charter invariant or the authority boundary is a breaking
Charter change and requires the governed waiver and human review flow.
